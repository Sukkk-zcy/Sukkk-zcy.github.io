## 一、OpenFlow 版本概览

在上篇（SDN 三层架构）中我们已经介绍了 OpenFlow 的版本演进，这里直接进入核心机制。目前学术界和工业界主要使用 **OpenFlow 1.3**，这也是我们学习的重点。

## 二、流表（Flow Table）

流表是 OpenFlow 交换机的核心。每条流表项定义了一条转发规则，告诉交换机"遇到什么样的包，怎么处理"。

### 2.1 流表项的结构

一条流表项由四个部分组成：

| 组成部分 | 作用 |
|----------|------|
| 匹配字段（Match Fields） | 定义匹配条件：源/目的 MAC、IP、端口号等 |
| 优先级（Priority） | 多条规则都能匹配时，优先级高的赢 |
| 计数器（Counters） | 记录匹配到的包数量和字节数 |
| 动作集（Instructions/Actions） | 匹配后执行的操作 |

### 2.2 匹配字段详解

OpenFlow 1.0 支持 12 个匹配字段，OpenFlow 1.3 扩展到了 40 多个。常用的包括：

```
入端口 (Ingress Port)
─────────────────────
源 MAC 地址 (Eth Src)
目的 MAC 地址 (Eth Dst)
以太网类型 (Eth Type)
VLAN ID
VLAN 优先级
─────────────────────
源 IP 地址 (IP Src)
目的 IP 地址 (IP Dst)
IP 协议号 (IP Proto)
IP DSCP
IP ECN
─────────────────────
TCP/UDP 源端口 (Transport Src)
TCP/UDP 目的端口 (Transport Dst)
```

这些字段可以任意组合使用。比如你可以只匹配"目的 IP + TCP 端口"，也可以匹配"入端口 + 源 MAC + VLAN ID"，灵活度很高。

### 2.3 通配符

不是所有字段都需要精确匹配。OpenFlow 支持通配符：

- **精确匹配**：`192.168.1.1` — 只匹配这一个 IP
- **前缀匹配**：`192.168.1.0/24` — 匹配整个网段
- **全匹配**：`*` — 任何值都匹配

## 三、流表匹配流程

这是 OpenFlow 最核心的机制。当一个数据包进入交换机时，处理流程如下：

### 3.1 完整流程图

```
数据包进入交换机
       |
       v
  ┌─────────────┐
  │ 查找第 0 级流表 │  ← 从最低编号的流表开始
  └──────┬──────┘
         |
    ┌────┴────┐
    |         |
  匹配成功   未匹配
    |         |
    v         v
 执行动作   Packet-In
    |      发送给控制器
    |         |
    |         v
    |    控制器决策
    |    下发 Flow-Mod
    |         |
    v         v
  ┌─────────────┐
  │ 检查是否有     │
  │ Goto-Table 指令│
  └──────┬──────┘
         |
    ┌────┴────┐
    |         |
   有         无
    |         |
    v         v
 跳转到下一级  数据包
 流表继续匹配  离开交换机
```

### 3.2 关键概念：多级流表

OpenFlow 1.3 引入了**多级流表**，允许多张流表串联工作。数据包可以像过流水线一样，依次经过流表 0、流表 1、流表 2...

每一张流表执行特定功能：

| 流表 | 典型用途 |
|------|---------|
| 流表 0 | 首次分类：区分 ARP、ICMP、TCP/UDP |
| 流表 1 | 按 IP 网段做进一步路由决策 |
| 流表 2 | 按端口号做访问控制 |

通过 `Goto-Table` 指令，数据包可以在多级流表之间跳转。这种设计让复杂的转发逻辑可以拆分成多个简单步骤，每张表负责一部分。

### 3.3 匹配优先级

当一张流表中有多条规则都能匹配时：

1. **按 priority 值排序**：每条流表项有一个 priority 值，数值越大越优先
2. **相同 priority**：由交换机实现决定，OpenFlow 规范不保证具体顺序

注意：OpenFlow **不**按"匹配精确度"排序，只看 priority 数值。所以如果你想让更精确的规则先匹配，需要手动给它设更高的 priority。

## 四、动作集（Actions）

匹配成功后，交换机执行的动作。主要动作类型：

| 动作 | 说明 | 例子 |
|------|------|------|
| output | 从指定端口转发 | output:3 → 从端口 3 发出 |
| drop | 丢弃数据包 | 不指定 output 即为丢弃 |
| set-field | 修改包头字段 | 修改 VLAN ID、目的 MAC |
| push/pop-tag | 添加/移除标签 | 添加 VLAN 标签 |
| group | 转发到组表 | 用于广播、多播 |

一个流表项可以同时包含多个动作，按顺序依次执行。比如：先修改 VLAN 标签，再从指定端口转发。

---

## 五、组表（Group Table）

组表是 OpenFlow 1.3 引入的高级转发机制。当多个流表项需要执行相同的复杂动作时，可以把动作封装到组表中，流表项只需引用组 ID。

### 5.1 组表的四种类型

| 类型 | 用途 | 场景 |
|------|------|------|
| All | 所有端口都执行动作 | 广播、泛洪 |
| Select | 选择一个端口执行 | 负载均衡 |
| Indirect | 只有一个端口执行 | 高效复用 |
| Fast Failover | 优先执行活跃端口 | 链路冗余、故障切换 |

### 5.2 举个例子：广播

假设交换机要将一个 ARP 请求广播到所有端口。没有组表时，需要为每个端口写一条流表项。有了组表：

```
组表：Group ID = 1, Type = All
  → Bucket 1: output:1
  → Bucket 2: output:2
  → Bucket 3: output:3

流表项：match=* → group:1
```

只需一条流表项就能实现广播，大大简化了配置。

## 六、计量表（Meter Table）

计量表用于流量控制和 QoS（服务质量）。它可以对匹配的流量进行限速、优先级调度等操作。

### 6.1 工作原理

计量表不是独立的处理阶段，而是通过流表项的 `Meter` 指令引用的。当数据包匹配到某条流表项后，如果该流表项包含 Meter 指令，就会先经过计量表处理，再执行后续动作：

```
流表项匹配 → Meter 指令 → 计量表处理（限速/标记） → 执行动作
```

计量表可以：
- **限速**：超过阈值的包被丢弃或降级
- **标记**：修改 DSCP 字段，标记不同服务等级
- **统计**：统计各流量级别的包数量和字节数

### 6.2 典型场景

- 保证视频会议的带宽优先于文件下载
- 限制 P2P 下载的最高带宽
- 对不同租户的流量做差异化服务

## 七、控制器与交换机的通信

控制器和交换机之间通过 OpenFlow 消息通信，主要有三类消息：

### 7.1 控制器 → 交换机

| 消息 | 作用 |
|------|------|
| Features | 查询交换机能力（支持多少流表、端口数等） |
| Configuration | 设置交换机参数 |
| Modify-State | 增删改流表项（Flow-Mod） |
| Read-State | 读取流表、端口统计信息 |
| Packet-Out | 控制器通过交换机发送数据包 |

### 7.2 交换机 → 控制器

| 消息 | 作用 |
|------|------|
| Packet-In | 交换机遇到未知包，上送给控制器决策 |
| Flow-Removed | 流表项被删除时通知控制器 |
| Port-Status | 端口状态变化时通知控制器 |
| Error | 报告错误 |

### 7.3 对称消息

| 消息 | 作用 |
|------|------|
| Hello | 建立连接时互相打招呼 |
| Echo | 心跳检测，确认连接存活 |
| Experimenter | 厂商自定义扩展 |

### 7.4 一次完整的交互流程

```
1. 交换机启动，与控制器建立 TCP 连接
2. 双方交换 Hello 消息，协商 OpenFlow 版本
3. 控制器发送 Features-Request，查询交换机能力
4. 交换机回复 Features-Reply，报告支持的流表数、端口等
5. 控制器通过 Modify-State 下发初始流表规则
6. 数据包到达，交换机按流表转发
7. 遇到未知包 → Packet-In → 控制器决策 → Packet-Out / Flow-Mod
```
